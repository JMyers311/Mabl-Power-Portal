Best Practices Standards
🧩
Test Design
Writing maintainable, reliable tests
  • Reuse Flows — extract common actions (login, navigation) into shared flows
  • Data-driven variables — use DataTables and Plan-level variables; never hard-code test data
  • Meaningful assertions — avoid "element exists"; assert on expected values and states
  • Element selection — prefer id and data-* attributes; avoid auto-generated XPath
  • Conditional logic — handle dynamic paths (e.g., skip login when session already active)

Naming & Branching Standards
Consistency across your workspace
📌 Use the master branch for all new tests unless versioning specifically requires a branch.
  • WIP prefix: name tests as WIP – [App] – [Feature] – [Test]
  • Labels: tag WIP tests; disable until reviewed; add to plans only when finalized
  • Status tracking: move tests through review → publish before including in CI/CD plans
  • Jira linking: link tests to work items to maintain full traceability
⚠️
Anti-patterns to Avoid
Common mistakes that cause flakiness
  • Don't hard-code test data — use data-driven approaches for flexibility and reuse
  • Don't UI-test what can be validated at the API layer — API tests are faster and more reliable
  • Don't add waits first — adjust Unified Runner interaction speed before resorting to explicit waits
  • Avoid auto-generated XPaths — retarget to stable attributes when selectors break
  • Don't skip documentation — all test artifacts must be reviewed and approved; use only validated tools

Troubleshooting Support
⚠️ Before adding waits: always adjust the Unified Runner page interaction speed first. Explicit waits should be a last resort.
🔧
Common Issues & Fixes
Click to expand each topic
  • Retarget to stable attributes — prefer id, data-testid, or aria-label
  • Avoid auto-generated XPaths — they break when the UI re-renders
  • Review Mabl's auto-healing events in run results to identify selector drift
  • Use Visit URL for direct routing instead of repetitive click-through navigation
  • Reduce dependencies on previous page state when possible
  • Check for session or cookie interference between test runs
  • Adjust page interaction speed in Unified Runner settings before adding wait steps
  • For slow-loading elements, use Mabl's wait-for-element condition rather than fixed waits
  • Reference: Adjusting page interaction speed (Guide)
  • Confirm endpoint policies allow test runner traffic for apps behind VPN or private networking
  • Coordinate with Network owners to allowlist Mabl runner IPs where feasible
  • For API-gateway backed apps, verify authorization headers are correctly passed

Advanced Features Power Users
🤖
Auto-Healing Locators
  • Mabl's ML automatically adjusts locators when UIs change
  • Review healing events in run results to monitor drift and validate fixes
  • Periodically accept or reject healed selectors to keep tests clean
👁️
Visual Testing
  • Visual change detection catches unintended UI regressions automatically
  • Screenshots captured at each step for run evidence
  • Configure visual diff sensitivity per test or environment
📊
Performance Metrics & Dashboards
  • Track key timings per test run — identify slow pages and performance regressions
  • Monitor JavaScript errors and broken links caught during test execution
  • Use Mabl dashboards to review trends across plans, environments, and branches
